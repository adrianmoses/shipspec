# shipspec Methodology

shipspec is a spec-driven development workflow for building software with AI agents. It enforces a product engineering mindset — keeping the consumer of every feature in view — while producing structured artifacts that agents can read, execute against, and write back to reliably.

---

## Core Philosophy

**Specs are contracts, not documentation.** A spec written before implementation is a commitment. A decision record written after is an honest account of what was actually built. The gap between the two is where learning lives.

**Consumer first.** Every feature spec must answer: who is this for, what job does it do for them, and what would make them not use it. This applies even when the consumer is another system or agent. Bug fixes and maintenance operations are exempt.

**Agents need unambiguous inputs.** Prose is for humans. Specs optimized for agents use stable section headers, enumerated lists, constrained formats, and explicit required vs optional markers. Ambiguity in a spec becomes hallucinated compliance in an agent.

**Every operation produces an artifact.** No codeless spec, no specless code. Even lightweight operations — refactors, dependency updates — produce a record. This is what keeps the audit skill honest.

## Document Hierarchy

```
METHODOLOGY.md          ← this document
docs/specs/
  OVERVIEW.md           ← product vision, tech stack, testing suite
  ARCHITECTURE.md       ← technical architecture and system design
  ROADMAP.md            ← ordered feature list with status
  {NNN}-{feature}/
    spec.md             ← implementation plan (agent input)
    decision.md         ← post-implementation record (agent output)
docs/maintenance/       ← records for refactors, dependency updates, etc.
bugs/                   ← fix records
```

Numeric prefixes are stable identifiers. Feature directory names are human hints only — the canonical name lives inside the spec.

## Primary Paths

### 1. New Project + New Feature
The canonical path. All repo-level docs exist. Agent reads context, executes against `spec.md`, produces `decision.md`.

```
initialize → [OVERVIEW, ARCHITECTURE, ROADMAP] → spec.md → implement → decision.md
```

### 2. Existing Project + New Feature
Audit runs first to generate missing repo-level docs. Once complete, the project is on the canonical path.

```
audit → [OVERVIEW, ARCHITECTURE, ROADMAP] → spec.md → implement → decision.md
```

### 3. New Project + Bug Fix
Lightweight. No feature spec. Fix record captures what was broken, what changed, and whether a spec gap was exposed.

```
[bug reported] → fix → bugs/{id}-{description}.md
```

### 4. Existing Project + Bug Fix
Same as path 3. Audit is optional — trigger it if the codebase isn't understood well enough to fix safely.

```
[bug reported] → (audit?) → fix → bugs/{id}-{description}.md
```

## Utility Operations

These are maintenance skills that operate on existing specs or code without following a primary path.

| Skill | Purpose | Output |
|---|---|---|
| `revise-roadmap` | Reorder, add, or remove features | Updated `02-roadmap.md` |
| `revise-architecture` | Update when technical decisions drift | Updated `01-architecture.md` |
| `refactor` | Structural code change, no behavior change | `docs/maintenance/` record |
| `deprecate-feature` | Mark feature removed, close its decision record | Updated `decision.md` |

## Spec Status

All spec documents carry a `status` field in a frontmatter metadata table. Valid values:

| Status | Meaning |
|---|---|
| `draft` | In progress, not ready for agent execution |
| `approved` | Ready for agent execution |
| `inferred` | Generated by audit skill, not written from intent |
| `implemented` | Feature complete, decision record written |
| `superseded` | Replaced by a later spec, ref included |
| `deprecated` | Feature removed |

## Required vs Optional Sections

Spec templates mark sections as required or optional using an HTML comment on the header line:

```markdown
## Consumer Impact <!-- required -->
## Performance Considerations <!-- optional -->
```

Agents must populate all required sections. Optional sections may be omitted with a one-line reason.

---

## Decision Record Discipline

Decision records are agent output, not agent reflection. Templates ask only questions an agent can answer from its own execution trace:

- What was specified
- What was implemented
- What was skipped, and why
- What tests pass (with evidence)
- Did this expose any spec gaps

Agents must not be asked to assess whether a decision was correct. That is a human review concern.

---

## Testing

The testing suite is defined in `00-overview.md`. It is load-bearing — spec compliance is verified by tests, not by agent self-report. Decision records include a test evidence section that references actual test output, not a checkbox.
